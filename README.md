# 🦞 龙虾跨平台通信

> 让不同电脑上的 openclaw（龙虾）互发消息、分工协作。
> 靠163邮箱中转，不需要服务器，天然适配所有形态的龙虾（本地 / 沙盒 / 封闭云服务器 / 魔改阉割版）。
>
> **独有优势：**
>
> **1. 零部署、零服务器成本**
> - 不用买云服务器、不用搭 EMQX、不用维护中枢服务
> - 直接用现成 163 邮箱，开箱即用
>
> **2. 极端封闭环境全能穿透**
> - Minimax 封闭云、workbuddy 魔改、企业沙盒、内网隔离机
> - 只要能上网发收邮件就能通
> - 几乎没有任何防火墙会封 SMTP/IMAP/POP3
> - 不用长连接、不用端口映射、不用内网穿透
>
> **3. 天然离线消息、持久化、永不丢消息**
> - 龙虾离线关机，邮件一直在邮箱服务器
> - 上线自动拉取，天然消息队列、天然持久化
> - 比你自己搭 MQTT 还要稳的离线保活

---

## 一句话说明

把一只 AI 当「中枢」，给其他 AI（龙虾）派活，其他龙虾干完回报结果。全靠163邮件传递消息。

---

## 核心流程

```
中枢 ──── 派活 ────→ 干活龙虾
      ←─── 收到！ ──┤    （自动回复ACK）
      ←─── 结果！ ──┤    （龙虾干完主动发）
```

**自动机制：**
- 龙虾收到活 → 自动回复「收到了」
- 超时没回应 → 自动通知你「对方离线」（ack_timeout_min，默认60分钟）
- 龙虾收到活后一直不发结果 → 超时自动关链路（result_timeout_min，默认120分钟）

---

## 消息类型

| 类型 | 谁→谁 | 什么意思 |
|------|--------|----------|
| CMD | 中枢→龙虾 | 去干这个活 |
| ACK | 龙虾→中枢 | 收到了 |
| RESULT | 龙虾→中枢 | 干完了，结果在这 |
| ERROR | 龙虾→中枢 | 出错了 |
| EXPIRED | 中枢→龙虾 | 太久了，不用干了 |
| HELLO | 任意→全员 | 我上线了 |
| GOODBYE | 任意→全员 | 我下线了 |
| DISCUSS | 任意→参与者 | 讨论发言 |

## 部署（3步）

**第一步：装依赖**
```bash
cd ~/.workbuddy/skills/lobster-comm
npm install --production
```

**第二步：配置**
```bash
node scripts/lobster-comm.js setup
```
回答向导问题：邮箱地址、授权码、通信密钥、本龙虾名字、工作模式

**第三步：验证**
```bash
node scripts/lobster-comm.js test-conn
node scripts/lobster-comm.js status
```

## 日常用这些命令

**派活**
```bash
node scripts/lobster-comm.js send --to "龙二" --action "开票" --description "给张三开票"
```

**看有没有新消息**
```bash
node scripts/lobster-comm.js poll
```

**查看所有龙虾状态**
```bash
node scripts/lobster-comm.js status
```

## 多轮讨论

多只龙虾围绕一个话题展开多轮辩论，最终中枢给出结论。

### 角色分工

发起讨论前，先给每个参与者分配一个思维角色：

| 角色 | 思维方式 |
|------|----------|
| **找茬者** | 专门挑毛病、找漏洞 |
| **搜索者** | 必须搜索网络找证据才回答 |
| **细节控** | 纠细节、算数字、查一致性 |
| **发散者** | 天马行空、提可能性 |
| **务实者** | 只关心能不能落地、成本多少 |
| **归纳者** | 梳理各方观点、找共识 |
| **旁观者** | 不参与，只记录观察 |

### 讨论流程

```
第1轮
  ① 主持人（中枢）提出议题
  ② 所有参与者按角色各自发言

第2轮
  ③ 主持人总结第1轮所有人的观点
  ④ 主持人提出新问题，推进到第2轮
  ⑤ 所有参与者继续按角色发言

……（重复到最大轮次）

最后
  ⑥ 主持人发 CONCLUDE，宣布结论
```

### 主持人的职责

- **发起**：指定话题、参与者和角色
- **每轮总结**：等所有参与者发言完毕，主持人总结各方观点
- **推进轮次**：总结后开启下一轮
- **宣布结论**：讨论结束轮次达到上限时，给出最终结论

### 参与者的职责

- 按被分配的角色发言
- 收到讨论后必须回复（超时未回复会被系统标记轮次超时）
- 不用操心流程，只管按角色说话

### 发起讨论命令

```bash
node scripts/lobster-comm.js discuss \
  --topic "选哪个方案" \
  --to "龙二,龙三" \
  --roles "龙二:challenger,龙三:researcher" \
  --max-rounds 5 \
  --content "方案A还是方案B，请从各自角度分析"
```

- `--roles` 格式：参与者:角色，多个用逗号分隔
- Windows 用户必须用简写格式（`龙二:challenger`），不能用 JSON 格式

### 结束讨论

```bash
node scripts/lobster-comm.js conclude --thread "线程ID" --conclusion "最终采用方案B"
```

## 常见问题

| 问题 | 怎么办 |
|------|--------|
| 发了活没回应 | 超过ack_timeout_min（默认60分钟）没回就说明对方离线了 |
| 活发出去了但对方说没收到 | 对方可能离线，检查 status 里的在线状态 |
| 新龙虾不被发现 | 运行 `hello` 命令重新宣告上线 |
| 旧龙虾还在列表里 | 运行 `forget 龙虾ID` 移除 |
| 收到活之后怎么做 | 干完执行任务，然后发 RESULT 或 ERROR 回报即可 |

---

## 注意

- **轮询间隔**不要太短，5分钟以上比较稳
- **邮件有延迟**，秒级实时不适合这个方案
- **通信密钥**所有龙虾必须一致
- **Windows 用户**：讨论角色用简写格式 `--roles "龙二:challenger"`，不要用JSON格式
